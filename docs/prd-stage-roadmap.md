# AI 外呼平台阶段性 PRD

Created: 2026-07-06

## 1. 文档目的

本文档基于当前 `ai-call` 项目能力、已有架构文档和外部市场调研，定义 AI 外呼平台从 PoC 到可试点、可运营、可规模化的阶段性产品需求。

当前项目已经具备外呼任务、流程编排、场景配置、知识库占位、音色克隆、全局规则、RBAC、通话记录、Voice Agent 和 FreeSWITCH 调度基础。
下一阶段的产品重点不应停留在“能打电话”，而应升级为“能批量运营、能合规、能评估效果、能接入业务系统”的外呼运营平台。

## 2. 产品定位

AI 外呼平台面向有高频电话触达需求的业务团队，帮助他们把催收提醒、售后回访、预约邀约、通知确认等电话工作流自动化，并在人工可控、合规可审计、效果可量化的前提下提升运营效率。

### 2.1 核心价值主张

- 对运营团队：从单次任务管理升级为活动级外呼运营，支持名单、排期、重拨、转化漏斗和导出。
- 对业务主管：实时看到接通率、转化率、失败原因、客户意向和话术效果。
- 对质检/合规人员：识别敏感话术、投诉风险、AI 身份披露缺失、黑名单或频控违规。
- 对实施/开发人员：通过知识库、工具插件、Webhook 和 CRM 集成把外呼接入真实业务流程。
- 对人工坐席：接收高意向或高风险客户的转人工任务，并获得通话摘要和上下文。

### 2.2 当前项目基础

| 能力 | 当前基础 | 后续产品化方向 |
| --- | --- | --- |
| 外呼任务 | 已有任务创建、批量导入、状态和通话尝试 | 升级为 Campaign 活动运营 |
| 流程编排 | 已有 TaskFlow、发布版本和节点执行 | 增加发布前测试、A/B 和流程分析 |
| 场景配置 | 已有催收、电商售后、售前邀约 | 增加行业模板和场景测试记录 |
| 通话记录 | 已有 TranscriptTurn、CallEvent、CallAttempt | 增加 Post-call Analysis 和质检结论 |
| 知识库 | 页面和 Mock 检索已存在 | 补齐上传、向量化、检索测试和版本绑定 |
| 全局规则 | 已有外呼时间窗、黑白名单、频控雏形 | 拆成合规中心和审计中心 |
| 工具调用 | 已有 mock 工具和 API 插件配置 | 产品化为集成中心和插件市场 |
| 人工介入 | Voice Agent 和 API 有转人工事件基础 | 建立坐席队列、处置结果和回拨闭环 |

## 3. 市场痛点与调研结论

| 市场痛点 | 用户真实问题 | 产品机会 |
| --- | --- | --- |
| 批量运营难 | 用户不是只打一通电话，而是管理一批名单、一场活动和多个批次 | 建设 Campaign 活动模块 |
| 效果不可见 | 接通、拒绝、转化、转人工和失败原因散落在通话记录里 | 建设 Analytics 和 Post-call Analysis |
| 合规风险高 | AI 外呼涉及授权、黑名单、频控、拨打时间窗、AI 身份披露和投诉追踪 | 建设 Compliance Center |
| 话术迭代慢 | 发布流程前缺少回归测试，业务不知道改动是否提升转化 | 建设场景测试、流程仿真和 A/B 对比 |
| 知识不可控 | RAG 命中了什么、为什么回答、文档是否过期都不透明 | 建设知识库闭环和引用追踪 |
| 业务系统割裂 | 外呼结果需要进入 CRM、短信、工单、订单、回访系统 | 建设 Integrations 和工具插件 |
| 人工承接断层 | AI 判断要转人工后，坐席没有上下文，也无法形成后续任务 | 建设 Handoff 工作台 |
| 生产稳定性不可观测 | 延迟、成本、Provider 错误、FreeSWITCH 状态和模型调用失败缺少统一视图 | 建设 Observability 和成本中心 |

### 3.1 外部参考

- [Vapi Outbound Campaigns](https://docs.vapi.ai/outbound-campaigns/overview)：竞品已将外呼活动、名单、排期、变量、通话记录作为标准外呼能力。
- [Retell Post-call Analysis](https://docs.retellai.com/features/post-call-analysis-create)：竞品强调将通话转成摘要、成功判断、自定义字段提取等结构化结果。
- [Vapi Knowledge Base](https://docs.vapi.ai/knowledge-base)：语音 Agent 平台普遍把知识库作为降低幻觉和提升业务准确性的核心模块。
- [Vapi Tools](https://docs.vapi.ai/tools/)：工具调用是语音 Agent 进入真实业务系统的关键能力。
- [FCC AI voice declaratory ruling](https://docs.fcc.gov/public/attachments/FCC-24-17A1_Rcd.pdf)：AI 生成语音电话在美国会被纳入 TCPA 对人工/预录语音电话的监管框架。
- [人工智能拟人化互动服务管理暂行办法](https://www.cac.gov.cn/2026-04/10/c_1777558395078289.htm)：中国市场也在强化 AI 拟人化互动服务治理，产品需要提前保留合规和审计能力。

## 5. 阶段路线总览

## 6. Phase 1 PRD：可试点外呼运营闭环

### 6.1 阶段目标

让业务团队可以围绕一次外呼活动完成名单导入、批次执行、效果查看、通话质检和合规控制。
该阶段的成功标准是：一个非技术运营人员可以独立创建一次外呼活动，并在活动结束后获得结构化业务结论。

### 6.2 范围

| 模块 | 页面 | 优先级 | 说明 |
| --- | --- | --- | --- |
| 外呼活动 | `/campaigns`、`/campaigns/new`、`/campaigns/[id]` | P0 | 活动级管理名单、排期、批次、重拨和状态 |
| 效果分析 | `/analytics` | P0 | 按活动、场景、流程、时间查看核心指标 |
| 通话质检 | `/quality`、通话详情页新增质检 Tab | P0 | 自动摘要、意向、失败原因、合规风险 |
| 合规中心 | `/compliance` | P0 | 黑名单、白名单、拨打时间窗、频控、授权和审计 |

### 6.3 核心需求

#### 6.3.1 外呼活动 Campaign

- R1. 运营人员可以创建外呼活动，填写活动名称、业务目标、场景、流程版本、拨打时间窗、并发上限和结束条件。
- R2. 运营人员可以上传 CSV/TSV/TXT 名单，并完成手机号校验、字段映射、变量预览、重复号码识别和无效行提示。
- R3. 系统应将活动拆分为可追踪的批次，展示待拨、拨打中、已接通、未接通、失败、已完成和已跳过数量。
- R4. 活动应支持草稿、已排期、运行中、已暂停、已完成、失败五类状态。
- R5. 运营人员可以暂停、恢复或终止活动，系统应保留操作人、时间和原因。
- R6. 活动应支持基础重拨策略，包括未接、占线、失败后的重试次数、重试间隔和最大拨打次数。
- R7. 活动详情应聚合通话尝试、客户变量、最终结果、最近一次通话摘要和下一步动作。
- R8. 活动结果应支持导出，至少包含客户号码脱敏值、客户姓名、状态、通话时长、业务结果、失败原因和转人工标记。

#### 6.3.2 效果分析 Analytics

- R9. 系统应展示活动漏斗：导入名单数、有效名单数、已拨打数、接通数、有效沟通数、目标达成数、转人工数、失败数。
- R10. 系统应支持按时间、活动、场景、流程版本和任务状态筛选指标。
- R11. 系统应展示接通率、平均通话时长、目标达成率、转人工率、失败率和平均重拨次数。
- R12. 系统应展示失败原因分布，例如无人接听、忙线、号码无效、客户拒绝、流程失败、合规拦截、系统异常。
- R13. 系统应展示流程节点掉点，帮助业务判断哪个话术或决策节点导致客户流失。

#### 6.3.3 通话质检与 Post-call Analysis

- R14. 每通电话结束后，系统应生成结构化分析结果，包括通话摘要、客户意向、业务结果、拒绝原因、下一步动作和置信度。
- R15. 系统应识别合规风险，包括未披露 AI 身份、敏感承诺、威胁性话术、频繁打断、客户明确拒绝后继续营销等。
- R16. 通话详情页应支持人工修正分析结果，修正内容应记录操作人和时间。
- R17. 质检列表应支持按风险等级、场景、活动、坐席转接、业务结果和时间筛选。
- R18. 质检结果应能反向用于活动统计和后续话术优化。

#### 6.3.4 合规中心 Compliance

- R19. 系统应提供统一合规策略配置，包括拨打时间窗、节假日规则、每日号码触达上限、活动并发上限和单号码最大拨打次数。
- R20. 系统应支持黑名单、白名单、退订号码和授权号码管理。
- R21. 系统应支持 AI 身份披露话术模板，并允许按场景绑定。
- R22. 系统应在拨打前执行合规拦截，并记录拦截原因。
- R23. 系统应记录合规相关审计日志，包括规则变更、名单导入、黑名单操作、活动启停和人工修正。

### 6.4 验收标准

- AE1. 给定一个已发布流程和 100 条客户名单，当运营人员创建活动并启动后，系统可以按时间窗自动拨打并展示活动进度。
- AE2. 给定名单中存在重复号码、无效号码和黑名单号码，当导入名单时，系统应在预览阶段提示原因，并允许下载错误明细。
- AE3. 给定一通已完成电话，当通话结束后，系统应在通话详情中展示摘要、业务结果、客户意向和合规风险。
- AE4. 给定客户明确表示“不再联系”，系统应将号码加入退订或待审核列表，并在后续活动中拦截。
- AE5. 给定业务主管查看活动分析页，系统应展示漏斗、趋势和失败原因，并支持按场景或流程版本筛选。

### 6.5 成功指标

| 指标 | 目标 |
| --- | --- |
| 活动创建耗时 | 首次使用不超过 10 分钟完成活动创建 |
| 名单导入成功率 | 有效格式文件导入成功率不低于 99% |
| 质检覆盖率 | 已完成通话 100% 生成结构化分析 |
| 合规拦截可解释率 | 每条拦截记录均有明确原因 |
| 活动结果可导出率 | 已完成活动 100% 可导出结果 |

### 6.6 非目标

- 不做复杂销售 CRM 的完整替代。
- 不做计费、套餐和商业结算。
- 不做多租户组织隔离。
- 不做跨渠道营销自动化。

## 7. Phase 2 PRD：可运营增长与业务集成

### 7.1 阶段目标

让外呼平台从“可以跑活动”升级为“可以持续优化业务结果”。
该阶段重点补齐知识库闭环、流程测试、业务系统集成、人工承接和更精细的重拨策略。

### 7.2 范围

| 模块 | 页面 | 优先级 | 说明 |
| --- | --- | --- | --- |
| 知识库闭环 | `/knowledge`、`/knowledge/[id]` | P1 | 上传、切片、检索测试、引用追踪、版本绑定 |
| 场景测试 | `/scenarios/[id]/tests`、流程调试面板 | P1 | 文本模拟、语音模拟、黄金测试集、发布前评分 |
| 集成中心 | `/integrations` | P1 | CRM、短信、Webhook、内部 API 和调用日志 |
| 人工承接 | `/handoffs` | P1 | 转人工队列、摘要、处置结果、回拨任务 |
| 策略引擎 | 活动详情和合规中心 | P1 | 更细的重拨、优先级和触达策略 |

### 7.3 核心需求

#### 7.3.1 知识库闭环

- R24. 用户可以创建知识库并上传文档，系统展示解析、切片、向量化和索引状态。
- R25. 用户可以在知识库详情页输入测试问题，查看召回片段、分数、来源文档和最终回答。
- R26. 场景或流程发布时应绑定知识库版本，避免运行时知识变化导致不可复现。
- R27. 系统应标记低置信度检索，并支持配置兜底话术或转人工策略。
- R28. 用户可以查看知识库命中统计，包括高频问题、无命中问题和过期文档候选。

#### 7.3.2 场景测试与流程仿真

- R29. 场景页的“测试记录”应展示每次测试的输入、模型输出、节点路径、命中知识和最终结果。
- R30. 流程发布前应支持文本模拟，用户可模拟客户回答并查看流程节点跳转。
- R31. 系统应支持黄金测试集，覆盖常见客户意图、拒绝理由、异常输入和转人工条件。
- R32. 流程发布前应给出测试通过率和高风险项提示。
- R33. 系统应支持对比两个流程版本在同一测试集上的表现差异。

#### 7.3.3 集成中心

- R34. 用户可以创建集成插件，配置名称、类型、Endpoint、鉴权方式、请求模板、响应映射和启用状态。
- R35. 系统应提供插件测试功能，用户输入样例变量后可以看到请求、响应和错误信息。
- R36. 系统应记录工具调用日志，包括来源通话、请求状态、耗时、错误码和重试次数。
- R37. 系统应支持基础 CRM 动作，例如创建线索、更新客户状态、记录通话摘要和创建回访任务。
- R38. 系统应支持短信动作，例如发送确认短信、退订确认和活动后跟进。

#### 7.3.4 人工承接 Handoff

- R39. 当通话触发转人工规则时，系统应创建转人工工单，包含客户信息、通话摘要、当前意图、风险标签和推荐动作。
- R40. 坐席可以查看待处理、处理中、已完成、已关闭四类工单状态。
- R41. 坐席可以填写处置结果，包括已联系、需回拨、无意向、已成交、投诉风险、号码错误等。
- R42. 坐席处置结果应回写活动分析和客户最终结果。
- R43. 系统应支持从 Handoff 工单创建回拨任务。

#### 7.3.5 策略引擎

- R44. 活动应支持按失败原因配置不同重拨策略。
- R45. 系统应支持按客户优先级、活动优先级和时间窗调度待拨队列。
- R46. 系统应支持号码级触达历史，避免多个活动重复触达同一客户。
- R47. 系统应支持策略模拟，启动活动前预估可拨数量和被合规拦截数量。

### 7.4 验收标准

- AE6. 给定一个知识库文档上传成功，当用户输入测试问题时，系统应展示召回片段和来源文档。
- AE7. 给定一个待发布流程，当黄金测试集中存在未覆盖转人工分支时，系统应阻止或警告发布。
- AE8. 给定一个 CRM Webhook 插件，当用户执行测试调用时，系统应展示请求体、响应体、耗时和错误。
- AE9. 给定一通触发转人工的电话，坐席应能在 Handoff 页面看到摘要和推荐动作，并填写处置结果。
- AE10. 给定同一号码出现在多个活动中，系统应按策略避免超频触达。

### 7.5 成功指标

| 指标 | 目标 |
| --- | --- |
| 知识库测试可解释率 | 每次测试均展示召回来源 |
| 发布前测试覆盖率 | 关键场景流程 100% 配置黄金测试 |
| 插件调用可追踪率 | 工具调用 100% 有日志 |
| 转人工处置闭环率 | 转人工工单 95% 有明确处置结果 |
| 重复触达拦截率 | 同号码跨活动超频触达 100% 可拦截 |

### 7.6 非目标

- 不建设完整客服工单系统。
- 不承诺覆盖所有第三方 CRM，只先支持通用 Webhook 和少量样例连接器。
- 不引入复杂机器学习训练平台。

## 8. Phase 3 PRD：规模化与商业化底座

### 8.1 阶段目标

让平台具备多团队使用、生产运维、成本核算、模板复用和持续优化能力。
该阶段适合从内部试点或作品集演示升级到可私有化部署或商业试点。

### 8.2 范围

| 模块 | 页面 | 优先级 | 说明 |
| --- | --- | --- | --- |
| 观测与成本 | `/observability`、`/costs` | P2 | Provider 延迟、错误率、通话成本和系统健康 |
| 模板中心 | `/templates` | P2 | 行业场景、流程、话术、知识库结构模板 |
| 多租户/组织 | `/organizations`、系统设置 | P2 | 组织隔离、配置隔离、权限隔离 |
| 数据闭环 | `/datasets`、`/insights` | P2 | 标注、拒绝原因聚类、话术优化建议 |
| 演示与交付 | `/demo-guide` 或首页引导 | P2 | 一键样例数据、演示流程、健康检查 |

### 8.3 核心需求

#### 8.3.1 观测与成本中心

- R48. 系统应展示 STT、LLM、TTS、FreeSWITCH、任务调度和工具调用的成功率、延迟和错误率。
- R49. 系统应展示每通电话的模型 Token、音频时长、Provider 成本和总成本估算。
- R50. 系统应支持按活动、场景、Provider 和时间查看成本趋势。
- R51. 系统应提供异常告警，包括 Provider 错误率升高、通话延迟异常、任务积压和 FreeSWITCH 连接异常。
- R52. 系统应支持导出运维报表。

#### 8.3.2 模板中心

- R53. 系统应提供行业模板，包括贷后催收、电商售后、试驾邀约、保险续保、教育回访、预约确认和通知提醒。
- R54. 模板应包含场景配置、流程图、推荐知识库结构、质检规则和指标建议。
- R55. 用户可以克隆模板并按业务修改。
- R56. 模板应标记适用场景、复杂度、推荐 Provider 和合规注意事项。

#### 8.3.3 多租户与组织

- R57. 系统应支持组织维度的数据隔离，包括活动、任务、流程、知识库、音色、插件和用户。
- R58. 系统应支持组织级 Provider 配置、合规策略和外呼额度。
- R59. 系统应支持组织管理员、运营、质检、坐席、开发者等角色。
- R60. 系统应提供组织级审计日志。

#### 8.3.4 数据闭环与持续优化

- R61. 用户可以将通话样本加入数据集，并标注意图、拒绝原因、合规风险和业务结果。
- R62. 系统应自动聚类高频拒绝原因和无命中知识问题。
- R63. 系统应基于数据集给出话术、知识库或流程优化建议。
- R64. 系统应支持对优化前后的活动进行指标对比。

#### 8.3.5 演示与交付体验

- R65. 系统应提供演示引导，帮助新用户按“创建场景、创建流程、导入名单、发起活动、查看结果”的路径体验产品。
- R66. 系统应提供样例数据重置能力，便于作品集展示、客户演示和本地测试。
- R67. 系统应展示部署健康检查，包括 API、Dashboard、Voice Agent、FunASR、FreeSWITCH 和数据库状态。

### 8.4 验收标准

- AE11. 给定一个活动完成后，业务主管可以查看该活动的总成本、平均成本和按 Provider 拆分的成本。
- AE12. 给定一个新用户进入模板中心，可以克隆“电商售后回访”模板并生成可编辑场景和流程。
- AE13. 给定两个组织用户登录，双方不能访问对方的任务、通话、知识库和插件。
- AE14. 给定一批已完成通话，系统可以聚合 Top 拒绝原因和无命中知识问题。
- AE15. 给定本地环境缺少 FreeSWITCH 连接，健康检查页应明确显示异常项和建议处理方式。

### 8.5 成功指标

| 指标 | 目标 |
| --- | --- |
| 生产问题定位耗时 | 常见 Provider 或调度问题 10 分钟内定位 |
| 模板创建活动耗时 | 通过模板创建首个活动不超过 5 分钟 |
| 组织数据隔离 | 跨组织数据访问 0 容忍 |
| 成本可归因率 | 95% 以上通话可归因到活动、场景和 Provider |
| 数据闭环使用率 | 重点活动 80% 以上有标注或洞察沉淀 |

## 11. 非功能需求

### 11.1 合规与安全

- 所有客户号码、通话文本和客户变量应按敏感数据处理。
- 导出文件应默认脱敏手机号，并记录导出人和导出时间。
- 合规策略变更必须可追溯。
- AI 身份披露、退订、黑名单、频控和授权记录应可审计。
- Webhook 和工具调用应支持鉴权、超时、重试和 allowlist。

### 11.2 稳定性

- 外呼调度、工具调用和分析任务应具备幂等能力。
- 活动暂停后不得继续发起新的外呼。
- Provider 调用失败时应可重试或降级，并记录失败原因。
- 通话事件和分析结果应可追踪到具体 CallAttempt。

### 11.3 可用性

- 活动创建流程应避免要求运营人员理解底层 Voice Agent 或 FreeSWITCH 概念。
- 列表页应提供搜索、筛选、导出和空状态引导。
- 风险提示应解释“为什么被拦截”或“为什么被标记”。
- 质检和分析结果应允许人工修正，不能只依赖模型判断。

### 11.4 可观测性

- 关键链路应记录结构化事件，包括活动调度、任务创建、呼叫发起、通话接通、转写、模型调用、工具调用、结束和分析。
- 所有失败应有可读错误原因和可检索错误码。
- 后续商业化应能按活动、组织、Provider 归因成本。

## 13. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 合规要求因地区和行业不同而变化 | 产品无法直接通用销售 | 将合规中心设计为策略配置和审计框架，不在产品中硬编码单一地区规则 |
| Post-call Analysis 误判 | 影响业务决策和质检可信度 | 引入置信度、人工修正和抽检机制 |
| 知识库回答不稳定 | 影响客户体验 | 引入引用追踪、低置信度兜底和发布版本绑定 |
| CRM 集成碎片化 | 实施成本高 | 先支持通用 Webhook，再沉淀高频连接器 |
| 活动并发导致系统压力 | 影响通话质量 | 引入活动级并发、调度队列和 Provider 健康监控 |
| 语音延迟或打断体验差 | 影响真实通话体验 | 在 Observability 中持续跟踪 STT/LLM/TTS 延迟和 barge-in 事件 |

## 14. 开放问题

- 目标市场优先选择国内私有化试点，还是海外 SaaS 形态。
- 是否需要在 Phase 1 就引入真实 CRM，还是先以 Webhook 和 Mock CRM 完成闭环。
- Post-call Analysis 使用同步生成还是异步任务生成。
- 合规中心是否需要支持多地区策略，例如中国、美国、欧盟分别配置。
- 多租户是否服务于商业化，还是仅用于演示多个客户环境。
